# Spec Kit Usage Guide

*[Português](spec-kit-guide.pt.md)*

Spec Kit is a set of skills that guides the feature-building process in a structured way, from concept to implementation. Each skill is one step in the pipeline.

## Full Pipeline

```
/specify → /clarify → /plan → /tasks → /analyze → /implement
                                                 ↑
                                           /converge (if stopped midway)
```

---

## File Structure

While working on a feature, spec-kit creates and maintains the following files under `specs/<feature-name>/`:

```
specs/
└── 001-feature-name/
    ├── spec.md          # Functional specification
    ├── plan.md          # Technical implementation plan
    ├── tasks.md         # Dependency-ordered task list
    ├── data-model.md    # Data model (generated by /plan)
    ├── research.md      # Technical research (generated by /plan)
    └── contracts/       # API contracts (generated by /plan)
```

The active feature is tracked in `.specify/feature.json`.

## Available Skills

### `/speckit-specify` — Specify the Feature

Creates or updates `spec.md` from a natural-language description.

```
/speckit-specify Add a card-balance lookup endpoint with Redis caching
```

**What it does:**
- Creates the feature directory under `specs/`
- Generates `spec.md` with context, functional and non-functional requirements, acceptance criteria and edge cases
- Updates `.specify/feature.json` with the active directory

**When to use:** At the start of any new feature, or to update requirements.

---

### `/speckit-clarify` — Clarify Requirements

Analyzes the current `spec.md` and asks up to 5 targeted questions to fill gaps, then folds the answers back into the spec.

```
/speckit-clarify
/speckit-clarify focus on authentication and rate limiting
```

**When to use:** After `/specify`, before planning. Avoids ambiguities that cause rework.

---

### `/speckit-plan` — Plan the Implementation

Generates the technical design artifacts based on `spec.md`.

```
/speckit-plan
/speckit-plan prioritize compatibility with the existing SQS queue architecture
```

**What it does:**
- Generates `plan.md` with architectural decisions, components and implementation strategy
- May generate `data-model.md`, `research.md` and files under `contracts/`

**When to use:** After clarifying the spec. This is the longest step.

---

### `/speckit-tasks` — Generate Tasks

Converts `plan.md` into a dependency-ordered task list in `tasks.md`.

```
/speckit-tasks
/speckit-tasks 20 tasks max, group by service
```

**What it generates:** Tasks with an ID, description, dependencies, affected files and completion criteria.

**When to use:** After `/plan`. A prerequisite for `/implement`.

---

### `/speckit-analyze` — Analyze Consistency

Checks consistency across `spec.md`, `plan.md` and `tasks.md` without changing anything.

```
/speckit-analyze
/speckit-analyze check that every spec requirement has a matching task
```

**What it checks:**
- Spec requirements with no tasks covering them
- Tasks with no connection back to the plan
- Circular dependencies or incorrect ordering

**When to use:** After `/tasks`, before `/implement`. Non-destructive — it only reports.

---

### `/speckit-implement` — Implement

Executes every task in `tasks.md` in order, marking each one complete.

```
/speckit-implement
/speckit-implement only the portador-service tasks
```

**What it does:** Reads each task, writes the code, runs tests where applicable, and marks the task `done`.

**When to use:** Once spec, plan and tasks have been reviewed and approved.

---

### `/speckit-converge` — Resume a Stalled Implementation

Compares the current state of the code against spec/plan/tasks and appends to `tasks.md` whatever parts are still unimplemented.

```
/speckit-converge
```

**When to use:** If `/implement` was interrupted midway, or if you made manual changes and want spec-kit to "catch up" to what's missing.

---

### `/speckit-checklist` — Custom Checklist

Generates a verification checklist specific to the current feature.

```
/speckit-checklist
/speckit-checklist focus on security and observability
```

---

### `/speckit-taskstoissues` — Export to GitHub Issues

Converts the tasks in `tasks.md` into GitHub Issues.

```
/speckit-taskstoissues
```

---

## Recommended Flow for a New Feature

```bash
# 1. Describe what you want to build
/speckit-specify Implement a card-blocking endpoint with SQS notification

# 2. Refine ambiguous requirements (optional but recommended)
/speckit-clarify

# 3. Plan the technical architecture
/speckit-plan

# 4. Generate the implementation tasks
/speckit-tasks

# 5. Check consistency across the artifacts
/speckit-analyze

# 6. Implement
/speckit-implement
```

---

## Flow for Resuming an Interrupted Feature

```bash
# See what's already done vs. what's missing
/speckit-converge

# Continue the implementation
/speckit-implement
```

---

## Tips

- **Each skill operates on the active feature** defined in `.specify/feature.json`. If you need to switch features, run `/speckit-specify` again with the other feature's name.
- **`/clarify` saves time**: the clearer the spec, the less back-and-forth during `/implement`.
- **`/analyze` is free**: it changes nothing, it only reports. Always run it before implementing.
- **You can edit `spec.md` and `plan.md` by hand** — spec-kit will respect the edited content in subsequent steps.
